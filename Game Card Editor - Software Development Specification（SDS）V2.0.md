# Game Card Editor
# Software Development Specification（SDS）

**Version：** V2.0（Simplified Edition）  
**Document Type：** Software Development Specification（SDS）

---

# 1. 项目概述

Global Kill Card Editor（全球杀卡牌编辑器）是一款基于 Web 的专业卡牌制作工具，用于快速完成卡牌的设计、编辑、管理与导出。系统定位为轻量化、可视化的图片与文字编辑器，而非游戏运行程序，也不负责管理游戏规则或业务逻辑。

编辑器的核心思想是：**所有卡牌都由固定尺寸画布上的多个图片图层与文字图层组合而成。** 用户通过模板创建卡牌，在画布中替换图片、编辑文字、调整图层位置与样式，最终合成为一张标准卡牌图片。

系统采用浏览器作为唯一运行环境，V2.0 优先支持本地部署运行，所有数据保存在本地 Workspace 中；后续可直接部署至服务器，无需修改整体架构。

---

# 2. 产品目标

本项目的目标不是开发一个通用图片编辑软件，而是开发一款专门面向卡牌制作场景的受限编辑器。系统应尽量降低学习成本，在满足卡牌制作需求的同时，避免引入与产品无关的复杂功能。

完整工作流程如下：

**创建项目 → 创建卡牌（或选择模板） → 上传图片素材 → 添加图片图层 → 添加文字图层 → 调整图层位置与样式 → 保存卡牌 → 导出 PNG → 自动生成缩略图 → 后续继续编辑**

整个过程遵循「所见即所得（WYSIWYG）」原则，编辑画布显示效果应与最终导出结果保持一致。

# 3. 产品范围

## 3.1 开发范围（In Scope）

V2.0 包含以下核心能力：

- 基于固定尺寸画布的卡牌编辑。
- 图片图层与文字图层编辑。
- 模板创建、上传与管理。
- 图片素材上传、管理与重复使用。
- 图层排序、锁定、隐藏及删除。
- 实时预览。
- Undo / Redo。
- 卡牌保存与继续编辑。
- PNG 高清导出。
- 自动生成缩略图。
- 本地 Workspace 管理。

## 3.2 非开发范围（Out of Scope）

V2.0 不包含以下功能：

- 游戏运行。
- AI 自动生成。
- 技能规则判定。
- 用户登录。
- 云端同步。
- 多人协作。
- 在线素材商城。
- 插件系统。
- 图片滤镜、钢笔工具等专业设计能力。

# 4. 系统架构

系统采用模块化架构，各模块保持低耦合、易维护。

| 模块 | 主要职责 |
|------|----------|
| Workspace Manager | 工作空间管理 |
| Project Manager | 项目管理 |
| Template Manager | 模板管理 |
| Asset Manager | 图片素材管理 |
| Canvas Editor | 画布编辑 |
| Layer Manager | 图层管理 |
| Property Panel | 元素属性编辑 |
| Export Engine | 图片导出与缩略图生成 |
| Persistence | 本地数据存储 |

# 5. 核心设计理念

编辑器内部不区分角色牌、武器牌、技能牌等业务对象，而是统一处理两种基础元素。

| 元素类型 | 说明 |
|----------|------|
| Image Layer | 所有图片，包括背景、边框、角色、图标、装饰等 |
| Text Layer | 所有文字，包括名称、说明、数值、标签等 |

对于编辑器而言，卡牌只是图片图层与文字图层按照一定顺序组合后的结果，业务含义由模板和用户决定，而不是由编辑器理解。

# 6. Workspace 与 Project

所有资源统一保存在 Workspace 中。

建议目录结构如下：

```text
Workspace/
├── Projects/
├── Templates/
├── Assets/
├── Exports/
└── Trash/
```

每个 Project 为一个独立项目。

```text
Project/
├── project.json
├── cards/
├── assets/
├── exports/
└── thumbnails/
```

Project 内可以包含多张卡牌，共享图片素材，并统一管理导出文件。

---

# 7. 模板系统

模板本质上是一份预先设计好的卡牌文档，而不是一套业务规则。

每个模板至少包含以下内容：

- 固定画布尺寸
- 默认图片图层
- 默认文字图层
- 图层顺序
- 图层默认属性
- 模板缩略图

建议目录结构如下：

```text
Template/
├── template.json
├── preview.png
├── background.png
└── assets/
```

模板支持上传、更新、删除和分类管理，可在多个项目之间重复使用。

---

# 8. 图片素材管理

图片上传后自动保存至当前 Project 的 Assets 中，并建立素材索引。

素材管理应支持：

- 上传
- 删除
- 重命名
- 分类
- 搜索
- 缩略图浏览
- 重复引用

同一张图片仅需上传一次，即可在当前项目内无限次使用。

---

# 9. 画布系统

画布是整个编辑器的核心。

每张卡牌对应一个固定尺寸画布，所有编辑操作均在画布范围内完成。

V2.0 建议支持以下能力：

| 功能 | 说明 |
|------|------|
| 固定画布尺寸 | 创建后保持固定 |
| 缩放查看 | 不改变实际尺寸 |
| 网格（可选） | 辅助排版 |
| 辅助线（可选） | 对齐参考 |
| 所见即所得 | 编辑结果等于导出结果 |

---

# 10. 图层系统

图层是编辑器唯一的编辑对象。

图片图层与文字图层均支持统一操作。

| 功能 | 图片 | 文字 |
|------|------|------|
| 选择 | ✓ | ✓ |
| 移动 | ✓ | ✓ |
| 缩放 | ✓ | ✓ |
| 旋转 | ✓ | ✓ |
| 锁定 | ✓ | ✓ |
| 隐藏 | ✓ | ✓ |
| 删除 | ✓ | ✓ |
| 调整层级 | ✓ | ✓ |
| 复制 | ✓ | ✓ |

编辑器不区分图层的业务含义，仅管理图层状态。

---

# 11. 属性面板

右侧属性面板根据当前选中对象自动切换。

| 当前对象 | 属性内容 |
|----------|----------|
| 未选择对象 | 画布属性 |
| 图片图层 | 图片、位置、尺寸、透明度等 |
| 文字图层 | 内容、字体、字号、颜色、对齐方式等 |

属性面板始终只显示当前对象的相关属性，避免复杂业务字段干扰编辑过程。

---

# 12. 数据模型

系统仅维护三类核心对象。

| 对象 | 说明 |
|------|------|
| Project | 一个完整项目 |
| Card | 一张卡牌 |
| Layer | 图片或文字图层 |

Layer 保存位置、尺寸、旋转、透明度、锁定状态及显示状态等信息，不保存业务逻辑。

---

# 13. 卡牌保存

每张卡牌保存完整编辑状态，而不仅仅保存最终图片。

建议结构如下：

```text
Card/
├── card.json
├── preview.png
├── export.png
└── history/
```

重新打开卡牌时，应恢复所有图层及编辑状态，实现无损继续编辑。

---

# 14. 导出规范

导出时自动生成两种图片。

| 文件 | 用途 |
|------|------|
| export.png | 高清正式输出 |
| preview.png | 卡牌缩略图 |

Project 首页默认读取缩略图进行展示，用户点击缩略图即可恢复对应卡牌继续编辑。

# 15. 开发阶段

项目采用渐进式开发，每个阶段形成可运行版本。

| 阶段 | 内容 |
|------|------|
| Phase 1 | Workspace、Project 基础框架 |
| Phase 2 | 模板系统与素材管理 |
| Phase 3 | 画布与图层系统 |
| Phase 4 | 图片与文字编辑 |
| Phase 5 | 保存与恢复 |
| Phase 6 | PNG 与缩略图导出 |
| Phase 7 | 测试、优化与发布 |

每个阶段均应完成设计、开发、测试和验收后，再进入下一阶段。

---

# 16. 开发规范

系统开发应遵循以下原则：

- 采用 TypeScript 严格模式。
- 保持模块职责单一，避免高耦合。
- 所有数据采用统一数据模型。
- 模板、素材、卡牌相互独立。
- 所有资源均通过统一管理器访问。
- 保持统一命名规范与目录结构。
- 新增功能必须同步完善测试。
- 架构设计优先考虑后续服务器部署，而不仅满足本地运行。
